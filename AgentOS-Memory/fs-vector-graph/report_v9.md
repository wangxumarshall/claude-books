# 架构持久化：大型语言模型Vector/Graph与Filesystem记忆范式的底层源码与工程实现深度研究

## 智能体记忆系统架构的演进与范式分化

在大型语言模型（LLM）从无状态文本生成器向具备长期自主性的智能体（Agent）演进的过程中，上下文窗口的物理限制与“迷失在中间”（Lost in the Middle）的注意力衰减效应构成了核心瓶颈。若缺乏外部化、持久化且可控的记忆架构，智能体将不可避免地陷入上下文截断导致的灾难性遗忘，无法在跨会话、跨任务的生命周期中保持逻辑连贯性与个性化状态 。

为突破此限制，当前的工程实践与学术研究已明确分化为两大底层记忆范式。

第一类为**Vector/Graph-like（向量与图计算）范式**，其核心思想是利用高维向量嵌入（Embeddings）捕捉语义相似度，并结合知识图谱（Knowledge Graphs）固化实体间的拓扑关系，从而模拟人类的联想记忆与语义记忆。该阵营的典型代表包括Letta（原MemGPT）、mem0、Graphiti/Zep、Honcho、eion、mem9以及UltraContext 。

第二类为**Filesystem-like（类文件系统）范式**，该范式彻底摒弃了纯向量检索的数学黑盒属性，转而将记忆、工具与知识映射为人类可读、结构高度决定论的操作系统目录与文件（如Markdown）。此类架构强调L0至L2的分层按需加载、影子索引（Shadow Index）机制以及递归目录检索，以OpenViking、memsearch、lossless-claw、Memoria、Voyager、Acontext及memU为典型代表 。

本报告将从源码级与论文技术细节层面，深度剖析这两大范式的核心子系统设计、数据流时序、并发控制以及Prompt组件装配逻辑。

## Vector/Graph-like范式：语义拓扑与动态推理体系

Vector/Graph-like范式将记忆视为一个流动的、相互连接的语义空间。它不依赖僵化的文件路径，而是利用向量数据库中的余弦相似度进行模糊匹配，利用图数据库进行多跳（Multi-hop）关系推理 。

### 项目概述与核心应用场景

该范式专为需要深度上下文综合、动态关系追踪与自主自我反思的复杂场景而设计。

- **Letta（MemGPT架构的演进）**：通过将智能体视作具有独立操作系统的状态机，致力于解决长期运行的个人助理与编码环境中的记忆分页问题 。
- **Graphiti与Zep**：面向企业级高并发部署，特别针对那些事实会随时间发生改变的场景，其内置的时态知识图谱（Temporal Knowledge Graph）能够完美处理“用户更换雇主”这类时效性事实冲突 。
- **mem0**：将自身定位为LLM应用的通用记忆层，其核心场景涵盖客户支持与医疗保健。它在架构上将向量数据库隐喻为“海马体”，将图数据库隐喻为“关联皮层”，从而实现从用户级、会话级到智能体级的多层次状态保留 。
- **Honcho**：提出了一种“记忆智能体”的全新概念，它在后台异步运行，不断从用户交互中提取显性与演绎性结论，构建深度的用户心理或行为画像（Peer Representation） 。
- **eion**：面向多智能体（Multi-agent）协同场景，通过共享关系图谱有效防止了智能体集群在复杂任务中的知识稀释与上下文漂移 。
- **Pancake**：从底层服务架构出发，专为多智能体LLM服务设计了基于动态CPU-GPU协同的缓存索引系统，解决了高并发下的记忆碎片化问题 。
- **mem9**：作为独立的记忆服务端，以插件形式介入智能体的执行生命周期。
- **UltraContext**：创新性地提供了一种类似Git的上下文API，允许开发者对无结构记忆状态进行分支（Fork）与时间旅行式的回滚，以应对复杂行为调试 。

### 记忆系统架构模型与多模态架构设计

在Vector/Graph范式中，原始输入字符串很少作为孤立的文本块被存储，它们必须经过多模态的解析、嵌入和拓扑映射。

#### Graphiti的三层子图架构

Graphiti的底层架构高度镜像了人类记忆的层级机制，由三个相互嵌套的子图（Subgraphs）构成：

1. **情景子图（Episodic Subgraph, $G_e$）**：最底层，负责持久化不可变的原始事件、消息与非结构化文档流。
2. **语义实体子图（Semantic Entity Subgraph, $G_s$）**：中间层，系统在此处通过提取管道识别实体（节点）及关系（边），并嵌入至1024维的向量空间中。
3. **社区子图（Community Subgraph, $G_c$）**：最顶层，利用标签传播算法对强连通实体进行聚类，并存储由LLM生成的宏观摘要，以供“鸟瞰式”检索 。

Graphiti最关键的设计是其**双时态模型（Bi-Temporal Model）**，它同时管理事件时间（现实中事实发生的时间）与摄取时间（系统学习到该事实的时间）。当事实发生变异时，系统不会物理删除旧边缘，而是对其打上invalid_at的时间戳，从而在保留历史溯源能力的同时保证检索的准确性 。

#### Letta的工作集架构

Letta的架构设计则围绕“工作集”（Working Set）理念展开，将记忆系统硬性划分为：
- **核心记忆（Core Memory）**：即LLM的上下文窗口，被进一步切分为多个功能性数据块（Blocks），如“角色设定”与“人类特征”。
- **归档记忆（Archival Memory）**：即外部的无限量向量存储。

模型不再是被动接收检索引擎灌入的上下文，而是被赋予了如core_memory_append或archival_memory_search等专用工具链，由LLM自主决定何时将信息换入或换出当前上下文 。

#### mem0的双模式架构

mem0的架构模型采取了显式的职责分离设计：
- 采用如text-embedding-3-small等模型生成的嵌入向量被持久化至Valkey、Pinecone或Qdrant等向量数据库中，以支持模糊的语义匹配；
- 显式的结构化关系（如“节点A-属于-节点B”）则并行流入Neo4j等图数据库后端。

这种**双模式（Dual-Schema）架构**有效弥补了纯向量存储无法进行有向关系推理的底层缺陷 。

### 子系统详细设计与底层实现

这类复杂架构的平稳运行，高度依赖于底层提取引擎、并发控制与压缩子系统的精密设计。

#### 推理、提取与核心引擎设计

在Graphiti中，语义提取子系统通过评估当前消息与前四个历史对话的滑动窗口来锁定命名实体。系统随后调用LLM执行共指解析（Coreference Resolution）逻辑，以决策是将新提及的对象实例化为全新图节点，还是将其特征合并至现有节点中 。

Honcho进一步深化了这一概念，其提取子系统完全基于异步的“**后台梦境（Background Dreaming）**”机制。当会话消息被摄入后，Honcho使用基于Qwen3-8B微调的Neuromancer XR推理模型，在后台自动剥离显性结论与演绎性结论。这种设计在不增加前端用户交互延迟的前提下，构建了极具深度的推理级记忆网络 。

#### 并发控制、数据库结构与多智能体同步

对于支持多智能体的系统，内存隔离与并发安全是设计的重中之重。

eion通过统一API封装了：
- PostgreSQL（集成pgvector处理对话历史与语义搜索）
- Neo4j（存储时态知识）

在数据库事务层，Neo4j的节点锁机制配合PostgreSQL的行级锁，确保了多智能体集群在同时读取共享上下文、更新知识图谱时不会产生脏读或竞争条件 。

Pancake则在近似最近邻（ANN）检索层面引入了高度动态的并发缓存子系统。由于传统的LSM树或周期性合并方法无法感知智能体工作负载中的请求内局部性（Intra-request Locality），Pancake设计了**三级聚类缓存（Three-Level Cluster Caching）**：

- **L0缓存**：维护了一个追踪最频繁访问微小聚类的转发表。
- 当特定智能体的L0聚类发生溢出时，驱逐策略会平滑地将不活跃向量迁移至底层存储，同时利用动态热点检测机制分配CPU与GPU协同计算资源，从而实现高达4.29倍的端到端吞吐量提升 。

#### 压缩引擎与生命周期管理

为了防止图数据库与向量索引的无限膨胀导致检索耗时呈指数级上升，生命周期管理子系统不可或缺。

Graphiti依赖异步的Map-Reduce作业定期遍历社区子图（$G_c$），生成并更新宏观摘要 。

Letta则实现了独特的“**休眠计算（Sleep-time Compute）**”机制。系统守护进程（Cron Job）会在无用户交互的闲置期唤醒智能体，指示其审查已经臃肿的核心记忆块，将零散事实进行碎片整理，并将冗余数据安全卸载至归档向量数据库中，实现了系统级别的自我修复 。

### 数据流时序图与控制流时序图

Vector/Graph架构的控制流具有高度的异步性或工具驱动性。

#### Graphiti/Zep的检索控制流

以Graphiti和Zep结合的典型检索控制流为例，其上下文装配的执行时序如下：

- **T0 (触发与预处理)**: 智能体发起对特定session_id或thread_id的内存读取请求，API网关接收请求并提取查询字符串。
- **T1 (混合检索阶段 $\phi$)**: 系统并发执行三条检索链路：
  - 链路A：对$G_s$图层执行余弦相似度匹配；
  - 链路B：基于BM25算法在事实字符串的属性字段上执行关键词检索；
  - 链路C：执行广度优先搜索（BFS），提取初始匹配节点周边预设跳数（Hops）的相邻边缘。
- **T2 (重排序阶段 $\rho$)**: 子系统运用倒数秩融合（Reciprocal Rank Fusion, RRF）算法，将语义评分、统计学关键词评分与拓扑连通性评分进行归一化并重新排序。
- **T3 (上下文装配阶段 $\chi$)**: 聚合排序最高的实体节点摘要、$G_c$层面的宏观群落摘要以及近期发生的$N$条原生原始消息。
- **T4 (提示词注入与返回)**: 系统将装配完成的数据序列化为人类可读的字符串格式，并返回给调用方，随后由智能体框架将其注入系统提示词中 。

#### Letta的工具调用循环

在Letta框架中，时序图体现为LLM与内存操作系统之间的工具调用循环：

1. 当消息到达时，LLM读取系统提示词与当前的核心内存状态；
2. 随后LLM可能生成一个带有thinking参数的工具调用，进行显式的思维链（Chain-of-Thought）推理；
3. 接着，LLM可能输出一个archival_memory_search工具调用以查询长期记忆；
4. 若工具调用结构中包含被设置为True的request_heartbeat参数，控制循环不会终止，而是将检索结果再次投喂给LLM，促使其进行连续的推理判断；
5. 直到LLM最终输出send_message指令并交回控制权 。

### System Prompt组件装配顺序

Vector/Graph范式对LLM系统提示词（System Prompt）的装配遵循严格的顺序，以确保基础行为准则与动态检出的记忆上下文互不干扰。

| 装配层级 | Letta (MemGPT OS架构) | mem9 / Zep (中间件注入架构) | Honcho (动态推理架构) |
|---------|----------------------|-----------------------------|----------------------|
| **第一级: 核心系统指令** | 不可变的底层架构规则，定义Persona、核心操作系统隐喻与状态机机制。 | 基础角色设定、目标准则。 | 核心角色、时态处理规则、显式与演绎逻辑执行规范。 |
| **第二级: 接口与工具契约** | 工具Schema的JSON定义（如core_memory_append及其输入输出限制）。 | 智能体可用工具集与挂载技能字典。 | 检索工具说明、多跳推理（Multi-hop）指引。 |
| **第三级: 状态元数据** | 系统性能指标（当前Token消耗、时间戳、上下文可用空间占比）。 | 会话元数据、当前绝对时间戳标记。 | 用户心理画像（Peer Representation）元数据。 |
| **第四级: 动态工作区** | 显式编辑的核心记忆块（Core Memory Blocks），受字符数严格截断控制。 | 中间件（如before_prompt_build钩子）动态注入的RRF排序后图谱上下文。 | 通过Dialectic API自动注入的脚手架式（Scaffolded）原子结论。 |
| **第五级: 情景缓冲区** | 滑动窗口截取的近期Raw对话历史序列。 | 当前Thread中最近发生的$N$条未经压缩的消息。 | 当前会话短时上下文。 |

在这种模式下，mem9这种插件体系表现出了极高的非侵入性。它注册监听诸如OpenClaw框架内的before_prompt_build等底层钩子事件。在提示词构建完成但发送给LLM之前的微秒级缝隙中，插件自动提取关键词、执行混合检索，并静默地将相关的向量/图谱记忆插入第四级位置，整个过程对智能体模型完全透明 。

## Filesystem-like范式：结构决定论与影子索引体系

与拥抱模糊匹配与隐式神经联想的Vector/Graph架构截然不同，Filesystem-like范式回归了计算机科学中最经典的文件树模型。它将记忆系统映射为确定性、具象化、人类完全可读的目录与Markdown文件，以牺牲部分并行匹配速度为代价，换取了绝对的系统可观测性与精准的Token控制。

### 项目概述与核心应用场景

该范式在以代码编写、复杂任务流编排为核心的AI编码智能体（AI Coding Agents）领域占据统治地位，因为在这些场景中，记忆系统的“幻觉”或不透明检索是致命的。

- **OpenViking**：构建了一个专为智能体设计的上下文数据库操作平台。它创新性地引入了viking://文件系统范式，将记忆、资源与执行技能全部纳入统一的树状目录结构中，彻底解决了传统RAG系统中文本块“碎片化”与“黑盒化”的痛点，显著降低了长上下文模型在投产时的Token消耗 。
- **memsearch**：作为一个轻量级的高性能守护进程，为Claude Code、OpenClaw等编码智能体提供跨平台的记忆持久化支持。它坚守“Markdown作为唯一事实来源”的原则，任何代理交互均被记录为.md文件，并在后台同步至向量加速引擎 。
- **lossless-claw**：针对主流智能体（如OpenClaw）自带的“滑动窗口丢弃”机制进行了革命性的重构。它不再粗暴地截断旧对话，而是将所有原生消息实时持久化至本地SQLite数据库中，并根据Token阈值，利用LLM将远端对话凝练为有向无环图（DAG），实现了长会话中绝对的零数据丢失与高保真召回 。
- **Memoria**：致力于层级工作记忆管理，其架构强制将个性化数据、浅层摘要与深层向量库严格划分为L0至L2三个物理加载层级，确保最核心的指令不会被长文本淹没 。
- **Voyager**：代表了具象化智能体在开放世界（如Minecraft）中的极致应用，其记忆系统表现为一个不断增长的、包含可执行代码的技能库（Skill Library），系统通过自动课程学习与迭代提示词机制，不断向文件系统中写入通过自我验证的复杂技能 。
- **Acontext**：作为一种面向自主学习智能体的上下文数据平台，将多模态工件与任务流状态直接映射存储，驱动智能体通过存储、观察、学习、行动（Store-Observe-Learn-Act）的循环进化 。
- **memU**：框架为实现24/7全天候在线智能体提供了基于使用频率而非单纯语义特征重组的轻量级主动记忆演化方案 。

### 记忆系统架构模型与影子索引设计

Filesystem范式的核心在于“结构即上下文”。传统RAG直接将文档打碎为无关联的Vector Chunks，而Filesystem架构保留了完整的目录语义和上下文边界。

#### OpenViking的协议化文件系统

OpenViking通过协议前缀强制将世界抽象化：
- `viking://user/memories/`：用于存放情节与用户偏好
- `viking://agent/skills/`：存放执行逻辑代码
- `viking://resources/`：存放外部知识实体

由于目录结构是显式的，LLM无需依赖向量检索寻找相关代码，而是可以直接发出类似`ls viking://agent/skills/`的目录遍历指令 。

为了兼顾快速定位与精确读取，系统严格推行**L0/L1/L2三层信息加载模型**：
- **L0层**：仅包含目录或文件的摘要（约100 Tokens）
- **L1层**：提供内容大纲与接口描述（约2000 Tokens）
- **L2层**：才是包含全量细节的原始文件

这保证了智能体在未确定目标前，绝不会被海量冗余数据撑爆上下文 。

#### memsearch的影子索引机制

memsearch的架构核心是“**影子索引（Shadow Index）**”机制。在这一设计中：
- 纯文本文件系统在处理大规模语义查询时效率极低
- 纯向量数据库又会导致“数据绑架”

因此，memsearch将位于`.memsearch/memory/*.md`的Markdown日志文件视为最高优先级的事实真相（Ground Truth），而将本地运行的Milvus Lite数据库视为一个可以随时被丢弃并完全重建的影子缓存。通过这种模式，用户获得了文件的透明可控性，同时保留了向量的毫秒级检索加速 。

#### lossless-claw的DAG架构

lossless-claw重塑了对话的缓存架构模型。系统不依赖外部知识库，而是完全聚焦于当前会话的无损压缩。

原始的交互流在存入SQLite后，当触发深度压缩阈值时，底部的旧消息被汇总为高维节点，新节点继续向后生长并关联，最终整个聊天历史在数据库中退化为一张复杂的有向无环图（DAG）。

通过专用的回调工具（如lcm_expand），模型可以沿着DAG的反向指针瞬间还原某一次具体操作的全部细节 。

### 子系统详细设计与底层实现

支持此类确定性架构需要特定的解析、同步去重以及递归探查子系统。

#### 影子同步与SHA-256去重引擎

以memsearch为例，其索引生命周期由多个精确的守护进程维系：

1. 当通过外部编辑器或智能体保存Markdown文件时，Watch子系统通过文件系统事件钩子捕捉变动。
2. 随后，执行性能最关键的Index工作流：
   - 系统首先对Markdown按逻辑段落进行智能分块（Smart Chunking）
   - 并为每个文本块计算一个唯一的SHA-256摘要哈希
   - 在将文本发送至Embedding模型（默认使用CPU友好的ONNX量化模型bge-m3-onnx-int8以节省成本）之前，系统会核对Milvus数据库中的哈希池
   - 未被篡改的段落会被直接跳过

这种原子级的变更检测，彻底消除了频繁文件保存带来的API开销与重计算延迟 。

#### 跨会话递归披露子系统 (Recursive Retrieval)

相较于Vector范式的混合计算打分，OpenViking探索出了一套高度自治的递归信息披露算法。该子系统的执行分五个严格阶段：

1. **意图分析（Intent Analysis）**：负责将长查询解构为子意图实体
2. **初始定位（Initial Positioning）**：系统调用轻量级向量搜素仅对顶级目录的L0元数据进行打分，圈定高潜目录
3. **精细探索**与**向下递归探索（Recursive Descent）**：智能体进入高分目录，动态获取内部子目录与文件的列表，如果当前层级信息不足以解答问题，模型将触发进一步的下钻指令，打开L2级别的明细文件
4. **结果聚合（Result Aggregation）**：该系统会生成一条完整的“检索轨迹（Retrieval Trajectory）”供开发者调试审计，彻底打破了传统黑盒 。
