### ✍️ 给 AI 的系统级写作要领 (Writing Guidelines for AI - 升级版)

在原有的基础上，必须向 AI 强调以下系统级约束，强化极客与学术严谨的双重属性：

1.  **基调约束：反过度包装 (Anti-Hype Tone)**
    *   **禁止使用**：涌现、意识、赛博生命、神奇、魔法等过度拟人化的词汇。
    *   **强制视角**：必须将 LLM 视为一个“具有概率性缺陷的文本生成函数”。整本书的基调是“如何用**类型系统与沙箱等硬强迫手段**管理这个不听话的函数”。
    *   **语气**：直接、犀利、极客风。像高级工程师在 Code Review 时一样一针见血。

2.  **结构约束：Show, Don't Tell (无代码不废话)**
    *   每个核心概念必须配备至少一个高阶代码块（如 Rust 的 `Result<T, Error>` 处理，TS 中 Mastra 的 `zod` 验证）。
    *   **对比驱动**：大量使用“反面教材（如：2023 年把所有 API 塞进 Prompt 导致模型崩溃）”与“正面教材（如：2026 年使用 RAG-MCP 动态检索 Top-K 契约）”的对比代码或表格。

3.  **核心学术/工程理念的植入 (Mandatory Concepts)**
    *   **CAR 模型贯穿全书**：在解释任何组件时，必须明确归类到 Control（控制/约束）、Agency（行动/工具）、Runtime（运行时/沙箱）这三层。
    *   **TNR 安全底线**：必须强调并定义**事务性无回归（TNR）**。不仅要有代码生成，必须包含发生异常时的回滚栈（Undo Stack）逻辑。
    *   **WASM/V8 隔离数据支撑**：在写第 8 章时，必须使用硬核数据说话（如：V8 Isolate 启动仅需几毫秒，内存占用是 Docker 的 1/100，利用 WASI 阻断非法 Fetch 调用）。

4.  **技术严谨度：拒绝伪代码**
    *   TypeScript 必须包含完整的类型推导（如泛型、Zod schema `zod.infer<typeof AgentState>`）。
    *   Rust 代码必须体现生命周期、所有权或宏的使用（例如 `ts-rs` 的跨语言绑定）。
